# Getting Started

From install to a running edit loop in about five minutes. By the end
you'll have a zero app on your machine, you'll know what each
generated file does, and you'll have made your first hand-edit.

## Install

zero ships as a single Rust binary. With a Rust toolchain on your
machine ([rustup.rs](https://rustup.rs) if you don't have one):

```sh
cargo install zero --locked
```

That's the whole install. No `node`, no `npm`, no global config.

Verify:

```sh
zero --version
```

## Scaffold a project

```sh
mkdir my-app && cd my-app
zero init
```

The first `zero init` runs an interactive wizard:

```
App folder name [web]:
Dev server port [3000]:
Backend proxy URL (leave blank for static SPA):
Build output folder [dist]:
```

Accept the defaults — you can change anything later by editing
`zero.toml`. To skip the wizard, write a `zero.toml` first and
`zero init` reads it directly:

```toml
[project]
root = "web"

[dev]
port = 3000

[build]
out = "dist"
```

The scaffold creates:

```
my-app/
├── zero.toml                # configuration (project, dev, build)
├── tsconfig.json            # editor-only TS settings (the CLI ignores it)
└── web/
    ├── index.html           # entry HTML; scripts are injected at serve time
    ├── src/
    │   ├── app.ts           # app entry — routes, state, mount
    │   └── routes/
    │       ├── home.ts      # the home route component
    │       └── home.test.ts # a starter test
    ├── styles/
    │   └── app.scss         # imports the zero design system
    └── .zero/               # auto-managed type defs, fonts, and components
                             # (commit it; `zero update` rewrites it)
```

`.zero/` is regenerated by `zero update` whenever you bump the CLI.
Treat it as managed: commit it so your editor sees the types, but
don't hand-edit anything inside.

## Start the dev server

```sh
zero dev
```

```
Listening on http://127.0.0.1:3000
```

Open the URL. You should see "Hello from zero" with a working
counter button.

The dev server transpiles `.ts` requests on the fly, serves
`/styles/*.scss` compiled to CSS, and falls back to `index.html`
for unknown paths so the router works on refresh. Every response
sends `Cache-Control: no-store` so a browser refresh always shows
the latest source. There is no HMR yet — a save triggers a full
reload.

If you set `[dev] proxy = "http://localhost:8080"` in `zero.toml`,
unknown paths forward to your backend instead of falling back to
`index.html`. That makes the browser and backend share one origin
during development, so no CORS configuration is needed.

## Anatomy of `src/app.ts`

```ts
import { App, signal } from "zero";
import Home from "./routes/home.ts";

const app = new App();
app.state("count", signal(0));   // app-level state — readable anywhere via inject()
app.route("/", Home);             // first match wins; add more routes here
app.run("#app");                  // mount into <div id="app"></div>
```

- `new App()` builds an empty app. `.state()`, `.route()`,
  `.layout()`, `.use()`, `.loading()`, `.error()` all return the
  same instance, so you can chain them.
- `app.state("count", signal(0))` registers an app-wide signal
  under the key `"count"`. Any component can read it with
  `inject("count")` — no prop drilling.
- `app.route("/", Home)` registers the home route. Routes match in
  the order you register; first match wins.
- `app.run("#app")` mounts. After this call the configuration is
  frozen: calling `.route()` or `.state()` again throws.

The full router story is in [Routing](./routing.html).

## Anatomy of a route component

```ts
import { html, inject } from "zero";
import type { Signal, TemplateResult } from "zero";

function Counter(): TemplateResult {
  return html`<p>Count: ${() => inject<Signal<number>>("count").val}</p>`;
}

export default function Home(): TemplateResult {
  return html`
    <main class="stack pad-xl align-center">
      <h1>Hello from zero</h1>
      <button @click=${() => inject<Signal<number>>("count").update(n => n + 1)}>
        Increment
      </button>
      ${Counter()}
    </main>
  `;
}
```

A few things worth noticing:

- **A component is a plain function** that returns the result of
  the `html` tagged template. It runs **once** when the framework
  commits it; the framework wires up granular reactive updates at
  every `${...}` site.
- **`html\`...\``** is a tagged template literal. Inside `${}` you
  can place strings, numbers, signals, arrays, nested
  `TemplateResult`s, or a function (a reactive block that re-runs
  when its dependencies change).
- **`@click=${...}`** binds a DOM event. Modifiers like
  `@click.prevent.stop=${...}` and key filters
  `@keydown.enter=${...}` are built in.
- **`inject<T>(key)`** reads app-level state by key. The generic
  parameter is purely a TypeScript hint — at runtime the value is
  whatever you registered.

If signals feel new, the next chapter — [Reactivity](./reactivity.html) —
walks through `signal`, `computed`, and `effect` from first
principles, with translations from React's `useState` /
`useMemo` / `useEffect`. The full vocabulary of `html` (attribute
binding, modifiers, `each`, `ref`) is in [Templates](./templates.html).

## Make the first edit

Let's replace the counter with a name field so we can watch
reactivity work.

Open `web/src/routes/home.ts` and replace it with:

```ts
import { html, signal } from "zero";
import type { TemplateResult } from "zero";

export default function Home(): TemplateResult {
  const name = signal("world");
  return html`
    <main class="stack pad-xl align-center">
      <h1>Hello ${name}</h1>
      <input
        value=${name}
        @input=${(e: Event) => name.set((e.target as HTMLInputElement).value)}
        placeholder="your name"
      />
    </main>
  `;
}
```

Save the file and refresh the browser. Type into the input — the
heading updates as you type. No re-render of the component, no
diffing: the framework knows the `${name}` substitution in the
`<h1>` is bound to the signal, and it patches that one text node
directly when the signal changes.

That's the whole reactive model in eight lines. The rest of the
docs are details.

## Build for production

```sh
zero build
```

This bundles your code into `./dist/`:

```
dist/
├── assets/
│   ├── app.<hash>.js
│   └── app.<hash>.css
├── manifest.json
└── index.html
```

`dist/index.html` is ready to upload to any static host. If you
have a backend, serve `dist/assets/` as static files and read
`dist/manifest.json` to inject the hashed script and link tags
into your server-rendered HTML.

Full coverage: [Building and Deploying](./building-and-deploying.html).
